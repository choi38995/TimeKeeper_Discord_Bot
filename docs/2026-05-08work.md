## 오늘 진행한 내용

오늘은 Discord 출석/퇴실 관리 봇에서 사용할 유저 상태 관리 로직과, 데이터를 GitHub private repository에 저장하는 방향에 대해 정리했다.

현재 직접 사용할 DB가 없는 상황이기 때문에, 별도의 GitHub private repository를 데이터 저장소처럼 사용하고, 봇에서는 해당 repo의 JSON 파일을 읽고 쓰는 방식으로 설계하기로 했다.

---

## 1. 유저 상태 관리 구조

처음에는 Java의 `List<Map<String, Object>>` 같은 구조를 Python에서 어떻게 표현할지 고민했다.

Python에서는 전체 유저 목록을 다음과 같은 딕셔너리 구조로 관리할 수 있다.

    users: dict[str, dict[str, object | None]] = {}

예상 데이터 형태는 다음과 같다.

    users = {
        "user_id_1": {
            "status": "offline",
            "reason": None
        },
        "user_id_2": {
            "status": "online",
            "reason": None
        }
    }

여기서 바깥쪽 딕셔너리의 key는 Discord 유저 ID이고, value는 해당 유저의 상태 정보를 담는 딕셔너리이다.

---

## 2. 딕셔너리 생성 의미 정리

처음에 헷갈렸던 부분은 다음 코드였다.

    users[user_id]["status"] = "offline"

이 코드는 `users[user_id]`가 이미 존재할 때만 가능하다.

새 유저를 추가할 때는 먼저 유저별 딕셔너리를 생성해야 한다.

    users[user_id] = {
        "status": "offline",
        "reason": None
    }

즉, "딕셔너리 생성"이란 유저별 기본 데이터 구조를 만들어 `users`에 넣는 것을 의미한다.

Java로 비유하면 다음과 비슷하다.

    Map<String, Object> user = new HashMap<>();
    user.put("status", "offline");
    user.put("reason", null);

    users.put(userId, user);

## 3. 주요 함수 설계

현재 고려한 주요 함수는 다음과 같다.

### initialize_users

봇 실행 시 또는 하루 시작 시 전체 유저 상태를 초기화한다.

    def initialize_users():
        if not users:
            print("No users to initialize")
            return

        for user in users.values():
            user["status"] = status.Status.CHECKOUT.value
            user["reason"] = None

        print("Initialized all users")

처음에는 `len(users) == 0`을 사용했지만, Python에서는 `if not users:`가 더 자연스럽다.

---

### add_users

유저 목록을 받아 아직 등록되지 않은 유저를 추가한다.

    def add_users(user_list: list[str]):
        for user_id in user_list:
            if user_id not in users:
                users[user_id] = {
                    "status": status.Status.CHECKOUT.value,
                    "reason": None
                }
                print(f"Added user {user_id}")

중요한 점은 `users[user_id]["status"] = ...`처럼 바로 접근하면 안 된다는 것이다.

새 유저라면 `users[user_id]` 자체가 아직 없기 때문에 `KeyError`가 발생한다.

---

### attendance_status

유저를 입실 상태로 변경한다.

    def attendance_status(user_id: str):
        if user_id in users:
            users[user_id]["status"] = status.Status.ATTENDANCE.value
            print(f"Updated {user_id} to {status.Status.ATTENDANCE.value}")
            return

        print(f"User {user_id} not found")

---

### checkout_status

유저를 퇴실 상태로 변경한다.

    def checkout_status(user_id: str):
        if user_id in users:
            users[user_id]["status"] = status.Status.CHECKOUT.value
            users[user_id]["reason"] = None
            print(f"Updated {user_id} to {status.Status.CHECKOUT.value}")
            return

        print(f"User {user_id} not found")

---

### away_status

유저가 입실 상태일 때만 자리비움 상태로 변경한다.

    def away_status(user_id: str, reason: str):
        if user_id not in users:
            print(f"User {user_id} not found")
            return

        if users[user_id]["status"] != status.Status.ATTENDANCE.value:
            print(f"User {user_id} is not {status.Status.ATTENDANCE.value}, cannot set to away")
            return

        users[user_id]["status"] = status.Status.AWAY.value
        users[user_id]["reason"] = reason
        print(f"Updated {user_id} to {status.Status.AWAY.value} with reason: {reason}")

Python에서는 `else if`가 아니라 `elif`를 사용한다.

또한 문자열 비교에는 `is not`이 아니라 `!=`를 사용해야 한다.

잘못된 예:

    users[user_id]["status"] is not "online"

올바른 예:

    users[user_id]["status"] != "online"

`is`는 값 비교가 아니라 객체 동일성 비교이기 때문에 문자열 비교에는 적합하지 않다.

---

### return_status

자리비움 상태인 유저를 다시 입실 상태로 변경한다.

    def return_status(user_id: str):
        if user_id in users and users[user_id]["status"] == status.Status.AWAY.value:
            users[user_id]["status"] = status.Status.ATTENDANCE.value
            users[user_id]["reason"] = None
            print(f"Updated {user_id} to {status.Status.ATTENDANCE.value} from away")
            return

        print(f"User {user_id} not found or not in {status.Status.AWAY.value} status")

## 4. Enum 사용 정리

Python에도 Java의 enum과 비슷한 기능이 있다.

    from enum import Enum

    class Status(Enum):
        ATTENDANCE = "online"
        CHECKOUT = "offline"
        AWAY = "away"

사용 예시는 다음과 같다.

    status.Status.ATTENDANCE.value
    status.Status.CHECKOUT.value
    status.Status.AWAY.value

다만 파일명과 클래스명은 일관성 있게 맞춰야 한다.

예를 들어 파일명이 `status.py`라면:

    import status

그리고 `status.py` 안에는 다음처럼 있어야 한다.

    from enum import Enum

    class Status(Enum):
        ATTENDANCE = "online"
        CHECKOUT = "offline"
        AWAY = "away"

Python 파일명은 보통 소문자와 언더스코어를 사용한다.

좋은 예:

    status.py
    user_status.py
    github_storage.py

비추천 예:

    Status.py
    UserStatusEnum.py

## 5. CHECKOUT vs OFFLINE 이름 고민

현재 enum은 다음처럼 되어 있다.

    class Status(Enum):
        ATTENDANCE = "online"
        CHECKOUT = "offline"
        AWAY = "away"

동작에는 문제가 없지만, 의미상 `CHECKOUT`은 행동이고 `OFFLINE`은 상태이다.

상태 enum이라는 관점에서는 다음이 더 자연스러울 수 있다.

    class Status(Enum):
        ATTENDANCE = "online"
        OFFLINE = "offline"
        AWAY = "away"

다만 현재 코드에서는 입실, 퇴실이라는 명령 흐름이 중요하므로 `CHECKOUT`을 사용해도 큰 문제는 없다.

중요한 것은 코드 전체에서 이름을 일관되게 사용하는 것이다.

## 6. reason 값 통일

초기 코드에서는 `reason` 값이 `""`와 `None`으로 섞여 있었다.

    "reason": ""

    users[user_id]["reason"] = None

둘 중 하나로 통일하는 것이 좋다.

추천은 `None`이다.

    "reason": None

이유가 없다는 의미를 더 명확하게 표현할 수 있기 때문이다.

## 7. GitHub private repository를 데이터 저장소로 사용하는 방향

현재 별도 DB를 사용할 수 없기 때문에, GitHub private repository에 JSON 파일을 저장하고, 봇에서는 GitHub API로 해당 파일을 읽고 쓰는 방식을 고려했다.

예상 구조:

    Discord Bot repo
    - work.py
    - status.py
    - github_storage.py
    - bot.py

    Private Data repo
    - users.json

`users.json` 예시:

    {
      "123456789": {
        "status": "offline",
        "reason": null,
        "attendance_time": null,
        "checkout_time": null
      }
    }

봇 동작 흐름:

    1. 봇 시작
    2. GitHub private repo에서 users.json 읽기
    3. Python dict로 변환
    4. 메모리의 users 딕셔너리에 저장
    5. 명령어 처리 시 users 수정
    6. 수정된 users를 다시 JSON으로 변환
    7. GitHub private repo의 users.json 업데이트

## 8. GitHub API 기본 개념

GitHub API는 GitHub 웹사이트에서 하는 작업을 코드로 수행할 수 있게 해주는 HTTP API이다.

파일을 읽을 때는 다음 API를 사용한다.

    GET /repos/{owner}/{repo}/contents/{path}

파일을 수정할 때는 다음 API를 사용한다.

    PUT /repos/{owner}/{repo}/contents/{path}

파일 수정 시에는 기존 파일의 `sha` 값이 필요하다.

따라서 저장 흐름은 보통 다음과 같다.

    1. GET 요청으로 현재 파일 정보 조회
    2. 응답에서 content와 sha 받기
    3. content를 base64 디코딩
    4. JSON을 Python dict로 변환
    5. dict 수정
    6. 다시 JSON 문자열로 변환
    7. base64 인코딩
    8. PUT 요청으로 GitHub에 업데이트

GitHub API를 사용하려면 토큰이 필요하다.

토큰은 코드에 직접 적지 않고 환경변수로 관리해야 한다.

나쁜 예:

    GITHUB_TOKEN = "ghp_xxxxx"

좋은 예:

    import os

    GITHUB_TOKEN = os.getenv("GITHUB_TOKEN")

## 9. GitHub Token 권한 설정

GitHub private repo의 `users.json` 파일을 읽고 쓰려면 fine-grained personal access token을 사용하는 것이 좋다.

추천 설정:

    Token type: Fine-grained personal access token
    Repository access: Only select repositories
    Selected repository: 데이터 저장용 private repo 하나
    Repository permissions:
    - Contents: Read and write
    - Metadata: Read

나머지 권한은 전부 `No access`로 둔다.

classic token도 가능하지만 private repo 접근을 위해 `repo` scope가 필요해서 권한이 너무 넓다.

따라서 지금 목적에는 fine-grained token이 더 적합하다.

## 10. 메모리 딕셔너리 캐싱

메모리 딕셔너리 캐싱이란 GitHub의 JSON 파일을 매번 읽지 않고, 봇 실행 중에는 Python 변수에 올려두고 재사용하는 방식이다.

현재 코드의 다음 전역 변수가 캐시 역할을 할 수 있다.

    users: dict[str, dict[str, object | None]] = {}

흐름은 다음과 같다.

    GitHub users.json
            ↓ 봇 시작 시 load
    Python users 딕셔너리
            ↓ 명령어 처리 시 수정
    Python users 딕셔너리
            ↓ 저장 시 save
    GitHub users.json

즉:

    GitHub users.json = 영구 저장소
    Python users dict = 실행 중 임시 저장소, 즉 메모리 캐시

예시:

    users = {}

    def load_cache():
        global users
        users = load_users_from_github()

    def save_cache():
        save_users_to_github(users)

주의할 점:

    봇이 종료되면 메모리 딕셔너리는 사라진다.
    GitHub에 저장하지 않은 변경사항은 유실될 수 있다.

따라서 중요한 변경 후에는 저장 함수를 호출해야 한다.

    attendance_status(user_id)
    save_users_to_github(users)

## 오늘 배운 핵심 요약

- Python의 딕셔너리는 key-value 구조이다.
- `users[user_id]["status"]`를 쓰려면 먼저 `users[user_id]`가 존재해야 한다.
- 새 유저 추가 시에는 유저별 기본 딕셔너리를 먼저 생성해야 한다.
- Python에서는 `else if`가 아니라 `elif`를 사용한다.
- 문자열 비교에는 `is not`이 아니라 `!=`를 사용한다.
- Python에도 Java의 enum 같은 `Enum`이 있다.
- `.py` 파일명은 보통 소문자와 언더스코어를 사용한다.
- GitHub private repo를 JSON 저장소처럼 사용할 수 있다.
- GitHub API로 파일을 수정할 때는 현재 파일의 `sha`가 필요하다.
- 토큰은 코드에 직접 쓰지 않고 환경변수로 관리해야 한다.
- 메모리 딕셔너리 캐싱은 봇 실행 중 Python 전역 딕셔너리에 데이터를 올려두고 재사용하는 방식이다.
